# PLAN: Implementation Review Agent

## User Assertions (verbatim, non-negotiable)

1. New agent and workflow job in `agentic-lib-workflow.yml` called **implementation-review**
2. Decompose elements of the mission
3. Trace each element through:
   - (a) Where implemented in `src/lib/`
   - (b) Where tested in unit tests
   - (c) Where tested in behaviour tests
   - (d) Where the website makes use of the feature
   - (e) How the website accesses the feature from the library
   - (f) How behaviour tests ensure the website presents the actual feature
   - (g) Any other useful checks that actual implementation is present — above what could be misleading metrics (e.g. "no open issues because they were closed in error" shouldn't look "finished")
4. Runs **concurrently with `maintain`** (both depend on `params` only)
5. Feeds into **director** and **supervisor** (both gain `implementation-review` as a dependency)
6. Writes a **table with each test and the result** to `agentic-lib-logs` branch
7. Writes **advice on completeness as an English sentence**
8. Each write to `agentic-lib-logs` should be to a **separate file** `agent-log-<datetime>.md` (e.g. `agent-log-2026-03-13T18-17-00-123Z.md`) — not appending to a single file
9. The write should follow **each `runCopilotSession()`** so we don't need to thread data through
10. During **telemetry**, gather all `agent-log-*.md` files from the logs branch
11. Place them on the **filesystem and make available to tools** in copilot sessions
12. **Supervisor** should examine the implementation review and **open an issue** if there is a gap
13. **Director** should take on board review comments when deciding **mission-complete, mission-failed, or in-progress**

## Goal

Add an `implementation-review` job that performs a structural trace of the mission through code, tests, website, and behaviour tests. This provides ground-truth evidence that the implementation actually exists — not just that metrics say it does. The review feeds into both supervisor (to open gap issues) and director (to gate mission-complete decisions), and all evidence is persisted as individual log files on the `agentic-lib-logs` branch for full auditability.

## Current State

- **10 task handlers** exist in `src/actions/agentic-step/tasks/` — none do implementation tracing
- **Logging** writes to a single `intentïon.md` file on the logs branch (append + rotate at 30 entries)
- **Telemetry** (`src/copilot/telemetry.js`) gathers metrics but doesn't read individual log files
- **Director** uses a deterministic metric assessment + LLM decision, with no structural trace input
- **Supervisor** reads the last 20 lines of the activity log — no structured review data
- **push-to-logs.sh** handles concurrent pushes to the orphan `agentic-lib-logs` branch

| `.github/workflows/agentic-lib-workflow.yml` | In the "Fetch log from log branch" steps (telemetry, director, supervisor, implementation-review), add a step that checks out the full `agentic-lib-logs` branch content into a staging directory (e.g. `.agent-logs/`) and lists all `agent-log-*.md` files. |
| `src/copilot/telemetry.js` | New function `gatherAgentLogs(logsDir)` that reads all `agent-log-*.md` files from a directory, parses their front matter, and returns structured data (task, outcome, datetime, test results table, completeness assessment). |
| `src/actions/agentic-step/tools.js` | Add a `list_agent_logs` tool and a `read_agent_log` tool so Copilot sessions can browse the log archive. The tools read from the `.agent-logs/` directory. |

### Workflow step pattern

```yaml
- name: Fetch agent logs from log branch
  run: |
    mkdir -p .agent-logs
    git fetch origin agentic-lib-logs 2>/dev/null || true
    # List all agent-log files on the branch and extract them
    for f in $(git ls-tree --name-only origin/agentic-lib-logs | grep '^agent-log-'); do
      git show "origin/agentic-lib-logs:${f}" > ".agent-logs/${f}" 2>/dev/null || true
    done
    echo "Fetched $(ls .agent-logs/agent-log-*.md 2>/dev/null | wc -l) agent log files"
```

### Success criteria

## Phase 3: Implementation Review Task Handler

_Requirements: 1, 2, 3a–3g, 6, 7_

### What

New task handler `implementation-review` that decomposes the mission into elements and traces each through code, tests, website, and behaviour tests.

### Files to create/change

| File | Change |
|------|--------|
| `src/actions/agentic-step/tasks/implementation-review.js` | **New file.** Task handler that: (1) reads MISSION.md, (2) reads features list, (3) uses Copilot session with read-only tools to trace each mission element through the codebase, (4) produces a structured review with test-result table and completeness assessment. |
| `.github/agents/agent-implementation-review.md` | **New file.** Agent prompt for the implementation review agent. |
| `src/actions/agentic-step/index.js` | Register `implementation-review` in the TASKS map. |

### Task handler design (`implementation-review.js`)

```javascript
export async function implementationReview(context) {
  // 1. Read mission, features, source tree
  // 2. Build prompt asking Copilot to decompose mission into elements
  // 3. For each element, trace through:
  //    - src/lib/ implementation (grep/read_file)
  //    - tests/ unit tests (grep/read_file)
  //    - tests/behaviour/ or playwright tests
  //    - website usage (if web path configured)
  //    - library→website integration path
  //    - behaviour test→website coverage
  // 4. Build structured output via report_implementation_review tool
  // 5. Return review data for logging

  const result = await runCopilotSession({
    model,
    agentPrompt: agentInstructions,
    userPrompt: buildReviewPrompt(mission, features, agentLogs),
    tools: [
      // Read-only tools: read_file, list_files, search_files, run_command (read-only)
      // Custom tool: report_implementation_review
    ],
    excludedTools: ['write_file', 'run_tests', 'dispatch', ...],
  });

  return {
    outcome: 'implementation-reviewed',
    reviewTable: result.reviewTable,       // structured test/trace table
    completenessAdvice: result.advice,     // English sentence
    gaps: result.gaps,                     // array of identified gaps
  };
}
```

### Agent prompt design (`agent-implementation-review.md`)

The agent should:
1. Parse the MISSION.md into discrete deliverable elements
2. For each element, use `list_files` and `read_file` to search for:
   - **Implementation**: Functions/classes in `src/lib/` that implement the element
   - **Unit tests**: Test files in `tests/` that import from `src/lib/` and test the element
   - **Behaviour tests**: Playwright/behaviour test files that exercise the element end-to-end
   - **Website usage**: Files in the web/website path that import or use the library feature
   - **Integration path**: How the website accesses the library (direct import, API call, etc.)
   - **Behaviour→website coverage**: Whether behaviour tests actually verify the website presents the feature
3. Flag **misleading metric patterns**:
   - Issues closed without a corresponding code change (closed-in-error)
   - Tests that exist but don't actually assert anything meaningful (empty/trivial tests)
   - Features listed as "done" in docs but missing from code
   - PRs merged without test coverage for the claimed feature
4. Report a `report_implementation_review` tool call with:
   - `elements[]`: Array of `{ name, implemented, unitTested, behaviourTested, websiteUsed, integrationPath, behaviourCoverage, notes }`
   - `gaps[]`: Array of `{ element, gapType, description, severity }`
   - `advice`: Single English sentence summarising completeness
   - `misleadingMetrics[]`: Array of `{ metric, reason, evidence }`

### Success criteria
- [ ] Task handler registered and callable via `agentic-step` action
- [ ] Agent decomposes mission into elements
- [ ] Each element traced through all 6 dimensions (3a–3f)
- [ ] Misleading metric patterns detected (3g)
- [ ] Output includes structured table + English completeness advice
- [ ] Agent log file written with full review data

## Phase 4: Workflow Job

_Requirements: 1, 4, 5_

### What

Add `implementation-review` job to `agentic-lib-workflow.yml` that runs concurrently with `maintain` and feeds into `director` and `supervisor`.

### Workflow changes

```yaml
  # ─── Implementation Review: traces mission elements through code/tests/website ──
  implementation-review:
    needs: [params]
    if: |
      !cancelled() &&
      (needs.params.outputs.mode == 'full' || needs.params.outputs.mode == 'maintain-only') &&
      needs.params.result == 'success'
    runs-on: ubuntu-latest
    outputs:
      review-advice: ${{ steps.review.outputs.completeness-advice }}
      review-gaps: ${{ steps.review.outputs.gaps }}
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
          ref: ${{ inputs.ref || github.sha }}

      - name: Fetch log and agent logs from log branch
        # ... fetch intentïon.md + all agent-log-*.md files

      - uses: actions/setup-node@v6
        with:
          node-version: "24"

      - name: Self-init (agentic-lib dev only)
        # ... same pattern as maintain

      - name: Install agentic-step dependencies
        # ... same pattern as maintain

      - name: Run implementation review
        id: review
        if: github.repository != 'xn-intenton-z2a/agentic-lib'
        uses: ./.github/agentic-lib/actions/agentic-step
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          COPILOT_GITHUB_TOKEN: ${{ secrets.COPILOT_GITHUB_TOKEN }}
        with:
          task: "implementation-review"
          config: ${{ needs.params.outputs.config-path }}
          instructions: ".github/agents/agent-implementation-review.md"
          model: ${{ needs.params.outputs.model }}

      - name: Push log to log branch
        # ... push intentïon.md + agent-log-*.md
```

## Phase 7: Log File Lifecycle & Cleanup

### What

Prevent unbounded growth of `agent-log-*.md` files on the logs branch.

### Design

- Keep the last **50** agent log files (configurable via `agentic-lib.toml`)
- During the telemetry step, after gathering logs, delete files older than the retention limit
- Push the cleanup as part of the normal log push

### Config addition (`agentic-lib.toml`)

```toml
[bot]
agent-log-retention = 50   # max agent-log files to keep on logs branch
```

### Success criteria
- [ ] Old log files are pruned during telemetry
- [ ] Retention limit is configurable
- [ ] Pruning doesn't lose the most recent logs

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Implementation review LLM session is expensive (many file reads) | Medium | Use read-only tools only, limit file content to first 100 lines per file, set a tool-call budget |
| Too many agent-log files slow down git operations on logs branch | Low | Phase 7 retention limit, orphan branch has no history |
| Review identifies spurious gaps (false positives) | Medium | Severity levels (critical/moderate/low), only critical gaps block mission-complete |
| Concurrent push collisions on logs branch | Low | Existing retry/rebase logic in push-to-logs.sh handles this |
| Website path not configured in some repos | High | Make website tracing optional — skip gracefully if `paths.web` is not set in config |
| Behaviour tests not present in all repos | High | Make behaviour tracing optional — report "not configured" rather than "gap" |
