## Gap Analysis: Why Isn't Mission Complete Declared?

All 7 acceptance criteria are satisfied. The code is correct, tested, and documented. The review agent confirmed this by closing issues #2676 and #2677 as RESOLVED. Yet the supervisor has not declared `mission-complete` after 14+ idle iterations.

### Perspective 1: My Assessment (Claude Code, External Observer)

The mission is unambiguously complete. The evidence:

1. **All acceptance criteria pass** — every criterion in MISSION.md has a corresponding passing test in `hamming.test.js`
2. **0 open issues** — nothing remains to be done
3. **2 recently-closed issues resolved by review** (#2676, #2677) — the review agent itself confirmed the code satisfies the requirements
4. **Unit tests pass** — CI test job succeeds on every single run
5. **README is documented** — CLI usage with examples
6. **Code quality is good** — Unicode-aware string handling, BigInt-safe bitwise operations, proper error types

There is zero gap between the current state and what MISSION.md requires. The `mission-complete` action should have been chosen no later than the iteration after #2676 was closed by review as RESOLVED (around 19:20 UTC). Instead, the pipeline has been burning compute for 3+ hours creating duplicate issues.

### Perspective 2: What the LLM Should Have Done (Expected Behaviour)

The supervisor prompt (agent-supervisor.md) explicitly defines when to declare mission-complete:

> **Mission complete signals:**
> - If the last 2+ issues were closed by review as RESOLVED, AND 0 open issues remain, the mission is likely accomplished. Verify against MISSION.md acceptance criteria, then use `mission-complete`.

The context sent to the LLM includes:
- **Recently Closed Issues**: showing #2676 and #2677 both "closed by review as RESOLVED"
- **Open Issues**: 0
- **MISSION.md**: with all acceptance criteria listed
- **Actions Since Last Init**: showing successful transforms and review closures

Given this context, the LLM should have:
1. Noticed 0 open issues
2. Noticed 2+ recently-closed issues resolved by review
3. Cross-referenced the acceptance criteria against the closed issues
4. Chosen `mission-complete | reason: All acceptance criteria satisfied — hammingDistance and hammingDistanceBits implemented with tests, CLI, docs, and README`

The prompt is clear and the evidence is unambiguous. The expected action is `mission-complete`.

### Perspective 3: What the LLM Actually Did

The LLM (gpt-5-mini) consistently chose `github:create-issue` to "Implement Hamming distance library" — the exact same work that is already done. Across 14+ runs:

- **10+ times**: Created a duplicate issue → caught by dedup guard → skipped
- **1 time**: Chose `nop` with sound reasoning ("workflows in progress, 0 open issues, wait")
- **0 times**: Chose `mission-complete`

The `nop` reasoning shows the LLM *can* read the context — it noticed "0 open issues" and running workflows. But it never connected "0 open issues" + "recently closed as RESOLVED" to the mission-complete criteria in its prompt.

**Root cause hypothesis**: The LLM appears to be anchoring on the mission description ("A JavaScript library exporting Hamming distance functions") and pattern-matching to "create an issue to implement it" without checking whether the implementation already exists. The recently-closed-issues section tells it the work is done, but the model's default behaviour is to create work rather than recognise completion. This suggests:

1. **The mission-complete detection logic is too passive** — it relies on the LLM choosing mission-complete from a menu of options. The LLM (gpt-5-mini specifically) has a bias toward creating issues rather than evaluating completion.
2. **The prompt may be too long** (19k+ input tokens) — the mission-complete criteria buried in a long prompt may get lost in context, especially for a smaller model.
3. **No source code in the prompt** — the supervisor sees issues, activity logs, and config but NOT the actual `src/lib/main.js`. It cannot directly verify that `hammingDistance` and `hammingDistanceBits` exist. It relies on the review agent's closure comments as indirect evidence.
4. **The "First supervisor run after init" bug** — every run incorrectly announces as the first run, which may be confusing the LLM into thinking this is a fresh start.

## Findings

### FINDING-1: Single-iteration implementation (POSITIVE)

Like fizz-buzz, the hamming-distance mission was fully implemented in a single transform. PR #2674 (issue #2672) created both `hammingDistance` and `hammingDistanceBits` with full validation, error handling, BigInt support, and tests in one 3-minute dev job. This is consistent with Report 001's FINDING-1 and confirms the pipeline can reliably go from empty template to working code in one cycle.

### FINDING-2: Code quality exceeds requirements (POSITIVE)

The generated code handles cases beyond the mission requirements:
- Unicode-aware string comparison (uses `Array.from()` for proper code point iteration)
- BigInt support in `hammingDistanceBits` (not required but useful for large integers)
- CLI interface with `string` and `bits` modes
- Proper error types (`TypeError` for wrong types, `RangeError` for invalid values)

### FINDING-3: "First supervisor run after init" fires on every run (BUG)

Every supervisor run logs "First supervisor run after init — announcing mission". The detection logic checks `ctx.recentActivity.includes("supervisor")` — if the activity log never contains the literal string "supervisor" from a prior run, every run thinks it's the first. This causes:
- Repeated bot dispatch attempts (all failing with "Unexpected inputs provided: ["message"]")
- Possible LLM confusion about whether this is a fresh mission or ongoing work

The bot dispatch failure ("Unexpected inputs provided: ["message"]") is a separate bug — the `agentic-lib-bot.yml` workflow doesn't accept a `message` input, so `dispatchBot()` always fails.

### FINDING-4: gpt-5-mini cannot detect mission completion (CRITICAL)

This is the central finding. The model consistently creates duplicate issues instead of declaring mission-complete, despite:
- Explicit prompt instructions with exact conditions
- 0 open issues in context
- 2+ recently-closed issues marked "RESOLVED" in context
- All acceptance criteria satisfiable from the evidence

The dedup guard prevents damage (issues get skipped), but the pipeline never converges to completion. This is a fundamental limitation of gpt-5-mini for the supervisor role — it lacks the reasoning depth to evaluate meta-conditions ("should I stop?") as opposed to task-oriented conditions ("what should I build?").

Comparison with fizz-buzz (Report 001): fizz-buzz declared mission-complete because the transform agent wrote `MISSION_COMPLETE.md` directly during transform #2 (PR #2669). The supervisor did NOT declare it — the transform handler did. This means the mission-complete path via supervisor has **never succeeded**.

### FINDING-5: Dedup guard prevents infinite issue creation (POSITIVE)

The dedup guard correctly catches duplicate issues by comparing new issue titles against recently-closed issues. Without this guard, the pipeline would create an unbounded number of identical issues. The guard saved ~10+ duplicate issues in this scenario.

### FINDING-6: Review agent correctly identifies completion (POSITIVE)

The review agent (review-issue task) correctly closed issues #2676 and #2677 as RESOLVED, confirming the code satisfies the issue requirements. The review agent's judgment is accurate — the code does implement everything. The gap is in the supervisor failing to use this signal.

### FINDING-7: Budget burn with no progress (CONCERN)

14+ idle iterations consumed budget units (maintain transformations, at minimum) without producing any code changes or mission-complete declaration. At 32 budget units, this rate of consumption will eventually exhaust the budget and trigger mission-failed rather than mission-complete — a perverse outcome for a mission that is actually complete.

### FINDING-8: Bot dispatch always fails (BUG)

Every supervisor run attempts `dispatchBot()` which calls `createWorkflowDispatch` with `inputs: { message: "..." }` on `agentic-lib-bot.yml`. The workflow does not accept a `message` input, so every dispatch fails with:
```
Could not dispatch bot: Unexpected inputs provided: ["message"]
```

This means no lifecycle announcements reach the discussions bot.

## Recommendations

### R1: Add deterministic mission-complete detection (HIGH PRIORITY)

Don't rely on the LLM to detect mission completion. After the LLM returns its actions, add a deterministic check:

```
if (0 open issues AND 2+ recently-closed as RESOLVED AND no transforms in last 3 runs):
    auto-execute mission-complete
```

This mirrors the dedup guard pattern — deterministic safety nets around LLM decisions.

### R2: Fix the "First supervisor run" detection (MEDIUM)

The `hasPriorSupervisor` check searches `ctx.recentActivity` for "supervisor", but the activity log may not contain this string. Fix by checking `ctx.actionsSinceInit` for supervisor workflow entries instead.

### R3: Fix bot dispatch inputs (MEDIUM)

The `dispatchBot()` function passes `{ message: "..." }` as workflow inputs, but `agentic-lib-bot.yml` doesn't accept this input. Either add the `message` input to the bot workflow or remove the dispatch call.

### R4: Test mission-complete with a stronger model (LOW)

Try the hamming-distance scenario with a larger model (e.g., claude-sonnet-4) as supervisor to determine whether the mission-complete detection failure is model-specific or prompt-structural.

### R5: Surface source code summary to supervisor (LOW)

The supervisor prompt includes issues, activity, and config but not the actual source code. Adding a summary of exported functions (e.g., "main.js exports: hammingDistance, hammingDistanceBits, main, usage") would give the LLM direct evidence that the implementation exists.

### R6: Declare this scenario complete manually

The hamming-distance mission is factually complete. Consider manually writing `MISSION_COMPLETE.md` and setting the schedule to off to stop budget burn, or implement R1 and let the deterministic check handle it.
